# Nabla 0.1 Release Candidate Checklist

Ce document fixe le perimetre et les criteres de sortie de Nabla 0.1. Il sert a
eviter l'empilement de fonctionnalites avant que le coeur du langage soit stable,
documente et teste.

## Objectif

Nabla 0.1 doit permettre d'ecrire de petits programmes idiomatiques avec une
syntaxe Scala-like, un typage statique simple et une compilation native Linux
x86-64 via NASM/ELF.

Le but n'est pas encore d'etre un langage generaliste complet. Le but est un
socle fiable : fonctions, classes, heritage, mixins/traits V1, collections de
base, lambdas, diagnostics lisibles et IO minimal.

## Surface incluse

### Langage

- Fonctions globales avec typage explicite.
- Variables locales `val` / `var`.
- Expressions `if` / `else if` / `else`.
- Boucles `while` et `for`.
- `match` simple avec wildcard final, gardes et motifs nommes simples.
- Classes avec constructeur primaire.
- Champs et methodes.
- Heritage nominal simple via `extends`.
- Mixins/traits V1 via `trait` et `with`, avec diagnostics de conflits.
- `override` obligatoire et signature strictement verifiee.
- `super` dans les methodes de classe.
- Hierarchie racine Scala-like `Any` / `AnyVal` / `AnyRef`.
- Boxing runtime minimal des primitives vers `Any` / `AnyVal` pour preserver
  `toString()` dynamique sur les appels de fonctions/methodes.
- Tag runtime dedie pour `String` afin de preserver `toString`, `hashCode` et
  `equals` par contenu apres effacement vers `Any`.
- Generiques simples invariants.
- Lambdas et types fonction `(A, B) => C`.
- Captures de lambda par valeur.

### Types de base

- `Unit`
- `Bool`
- `Int`
- `Long`
- `Float`
- `Double`
- `Char`
- `String`

### Stdlib publique recommandee

La surface utilisateur recommandee pour 0.1 est decrite dans
`docs/stdlib-api.md` et la reference HTML generee `docs/stdlib/`.

- `Array[T]` et le compagnon `Array` : `fill`, `range`, `rangeUntil`, operations
  communes verifiees par les cookbooks.
- `Option[T]` et le compagnon `Option` : `some`, `none`, `map`, `flatMap`,
  `foreach`, `filter`, `orElse`, `getOrElse`.
- `Set[T]` et le compagnon `Set` : `empty`, `fromArray`, operations immutables,
  predicats et conversions documentees.
- `Map[K, V]` et le compagnon `Map` : `empty`, `fromArray`, recherche
  optionnelle, mises a jour immutables, iterations et conversions documentees.
- `Sized` comme trait public minimal pour les collections exposant `size`,
  `isEmpty` et `nonEmpty`.
- Modules `strings`, `math`, `io` et `util` documentes.

Les anciens noms et representations (`ArrayFill`, `ArrayRange`, `ArrayObject[T]`,
`ArrayInt`, tableaux primitifs specialises, helpers `arrayBase...`, etc.) restent
possibles ou visibles dans certains retours/diagnostics, mais sont traites comme
compatibilite ou details de representation, pas comme surface recommandee.

### Backend et outillage

- Backend natif Linux x86-64.
- Generation NASM `elf64`.
- Edition de liens via `ld`.
- Diagnostics clairs quand `nasm` ou `ld` est absent.
- CI GitHub couvrant tests, exemples, tooling, docs stdlib generees, `-Werror`
  et espaces finaux.

## Non-objectifs pour 0.1

Ces sujets sont explicitement reportes :

- GC complet ou compactant.
- Backend autre que Linux x86-64 NASM/ELF.
- Unicode complet pour `String`/`Char`.
- Modules prives/export explicite au niveau langage.
- Traits/mixins avec linearisation Scala complete.
- Variance generique avancee.
- Fonctions generiques comme valeurs vraiment polymorphes.
- Typeclasses/implicites.
- Pattern matching exhaustif avance.
- `Result[T]` et collections riches au-dela de la surface deja documentee.
- Optimisations agressives.
- ABI publique stable.

## Checklist RC locale

Avant d'ouvrir ou de valider une RC, la branche `master` doit passer exactement :

```bash
PATH=/opt/data/local/usr/bin:$PATH make all-tests
PATH=/opt/data/local/usr/bin:$PATH make examples
PATH=/opt/data/local/usr/bin:$PATH make tooling-tests
PATH=/opt/data/local/usr/bin:$PATH make stdlib-docs
git diff --exit-code docs/stdlib
g++ -std=c++17 -Wall -Wextra -Werror \
  src/main.cpp src/parser.cpp src/ast.cpp src/semantic_analyzer.cpp \
  src/ir.cpp src/ir_codegen.cpp src/runtime_asm.cpp -o /tmp/nablac-werror
git diff --check
```

La commande `make stdlib-docs` doit etre suivie de `git diff --exit-code
docs/stdlib` pour confirmer que la reference HTML suivie est synchronisee avec
les commentaires `///`, `@signature` et `@symbol`.

## Checklist de contenu

- [x] `docs/language.md`, `docs/stdlib-api.md`, `docs/internals.md` et
  `docs/stdlib/` existent et decrivent la surface livree.
- [x] Les exemples publics sont verifies par `make examples`.
- [x] Les cookbooks publics privilegient `Array[T]`, `Option[T]`, `Set[T]`,
  `Map[K, V]`, `Sized` et les modules standards documentes.
- [x] Les diagnostics d'heritage/mixins, `override` et `super` ont des tests
  negatifs exacts.
- [x] Les diagnostics de compatibilite orientent vers les compagnons recommandes
  (`Array.fill`, `Array.rangeUntil`, `Set.fromArray`, `Option.some`,
  `Option.none`, etc.).
- [x] Les conventions runtime observables importantes sont documentees ou
  marquees comme internes.
- [ ] Lire une derniere fois la reference HTML publiee apres regeneration.
- [ ] Taguer `v0.1.0` seulement apres une validation verte sur `master` et une
  verification manuelle rapide des notes de release.

## Procedure de tag proposee

Apres merge de la PR de RC et validation verte sur `master` :

```bash
git fetch origin --prune
git checkout master
git pull --ff-only origin master
PATH=/opt/data/local/usr/bin:$PATH make all-tests
PATH=/opt/data/local/usr/bin:$PATH make examples
PATH=/opt/data/local/usr/bin:$PATH make tooling-tests
PATH=/opt/data/local/usr/bin:$PATH make stdlib-docs
git diff --exit-code docs/stdlib
g++ -std=c++17 -Wall -Wextra -Werror \
  src/main.cpp src/parser.cpp src/ast.cpp src/semantic_analyzer.cpp \
  src/ir.cpp src/ir_codegen.cpp src/runtime_asm.cpp -o /tmp/nablac-werror
git diff --check
git tag -a v0.1.0 -m "Nabla 0.1.0"
git push origin v0.1.0
```

## Politique de changements apres RC

Apres cette checklist, eviter toute nouvelle fonctionnalite visible avant
`v0.1.0`. Les changements acceptables sont :

- corrections de bug avec test cible ;
- corrections de diagnostic ;
- corrections de documentation ou d'exemples ;
- ajustements de release notes/checklist.

Toute extension de surface publique doit etre reportee apres `v0.1.0`, sauf si
elle corrige une incoherence bloquante de la surface deja documentee.
