# Nabla 0.1 Release Notes

Ce document décrit le périmètre livré par `v0.1.0` et conserve la matrice de
validation qui sert de référence pour les correctifs 0.1.x. Le tag `v0.1.0` est
présent localement et sur `origin`.

## Objectif

Nabla 0.1 permet d'écrire de petits programmes idiomatiques avec une syntaxe
Scala-like, un typage statique simple et une compilation native Linux x86-64 via
NASM/ELF.

Le but n'est pas encore d'être un langage généraliste complet. Le jalon fixe un
socle fiable : fonctions, classes, héritage, mixins/traits V1, collections de
base, lambdas, diagnostics lisibles et I/O minimal.

## Surface livrée

### Langage

- Fonctions globales avec typage explicite.
- Variables locales `val` / `var`.
- Expressions `if` / `else if` / `else`.
- Boucles `while` et `for`.
- `match` simple avec wildcard final, gardes et motifs nommés simples.
- Classes avec constructeur primaire.
- Champs et méthodes.
- Héritage nominal simple via `extends`.
- Mixins/traits V1 via `trait` et `with`, avec diagnostics de conflits.
- `override` obligatoire et signature strictement vérifiée.
- `super` dans les méthodes de classe.
- Hiérarchie racine Scala-like `Any` / `AnyVal` / `AnyRef`.
- Boxing runtime minimal des primitives vers `Any` / `AnyVal` pour préserver
  `toString()` dynamique sur les appels de fonctions/méthodes.
- Tag runtime dédié pour `String` afin de préserver `toString`, `hashCode` et
  `equals` par contenu après effacement vers `Any`.
- Génériques simples invariants.
- Lambdas et types fonction `(A, B) => C`.
- Captures de lambda par valeur.

### Types de base

- `Unit`
- `Bool`
- `Int`
- `Long`
- `Float`
- `Double`
- `Char`
- `String`

### Stdlib publique recommandée

La surface utilisateur recommandée pour 0.1 est décrite dans
`docs/stdlib-api.md` et la référence HTML générée `docs/stdlib/`.

- `Array[T]` et le compagnon `Array` : `fill`, `range(size)`,
  `range(start, until)`, `rangeUntil` en compatibilité et opérations communes
  vérifiées par les cookbooks.
- `Option[T]` et le compagnon `Option` : `some`, `none`, `map`, `flatMap`,
  `foreach`, `filter`, `orElse`, `getOrElse`.
- `Set[T]` et le compagnon `Set` : `empty`, `fromArray`, opérations immutables,
  prédicats et conversions documentées.
- `Map[K, V]` et le compagnon `Map` : `empty`, `fromArray`, recherche
  optionnelle, mises à jour immutables, itérations et conversions documentées.
- `Sized` et `Iterable[T]` comme traits publics minimaux pour les collections.
- Modules `strings`, `math`, `io` et `util` documentés.

Les anciens noms et représentations (`ArrayFill`, `ArrayRange`, `ArrayObject[T]`,
`ArrayInt`, tableaux primitifs spécialisés, helpers `arrayBase...`, etc.) restent
possibles ou visibles dans certains retours/diagnostics, mais sont traités comme
compatibilité ou détails de représentation, pas comme surface recommandée.

### Backend et outillage

- Backend natif Linux x86-64.
- Génération NASM `elf64`.
- Édition de liens via `ld`.
- Diagnostics clairs quand `nasm` ou `ld` est absent.
- CI GitHub couvrant tests, exemples, tooling, docs stdlib générées, `-Werror`
  et espaces finaux.

## Non-objectifs pour 0.1

Ces sujets restent explicitement reportés après le jalon 0.1 :

- GC complet ou compactant.
- Backend autre que Linux x86-64 NASM/ELF.
- Unicode complet pour `String`/`Char`.
- Modules privés/export explicite au niveau langage.
- Traits/mixins avec linéarisation Scala complète.
- Variance générique avancée.
- Fonctions génériques comme valeurs vraiment polymorphes.
- Typeclasses/implicites.
- Pattern matching exhaustif avancé.
- `Result[T]` et collections riches au-delà de la surface déjà documentée.
- Optimisations agressives.
- ABI publique stable.

## Matrice de validation 0.1.x

Avant de publier un correctif sur la ligne 0.1 ou de préparer un nouveau tag, la
branche `master` doit passer :

```bash
PATH=/opt/data/local/usr/bin:$PATH make all-tests
PATH=/opt/data/local/usr/bin:$PATH make examples
PATH=/opt/data/local/usr/bin:$PATH make tooling-tests
PATH=/opt/data/local/usr/bin:$PATH make stdlib-docs
git diff --exit-code docs/stdlib
g++ -std=c++17 -Wall -Wextra -Werror \
  src/main.cpp src/parser.cpp src/ast.cpp src/semantic_analyzer.cpp \
  src/ir.cpp src/ir_codegen.cpp src/runtime_asm.cpp -o /tmp/nablac-werror
git diff --check
```

La commande `make stdlib-docs` doit être suivie de `git diff --exit-code
docs/stdlib` pour confirmer que la référence HTML suivie est synchronisée avec
les commentaires `///`, `@signature`, `@status`, `@example` et `@symbol`.

## État de contenu

- [x] `docs/language.md`, `docs/stdlib-api.md`, `docs/internals.md` et
  `docs/stdlib/` décrivent la surface livrée.
- [x] Les exemples publics sont vérifiés par `make examples`.
- [x] Les cookbooks publics privilégient `Array[T]`, `Option[T]`, `Set[T]`,
  `Map[K, V]`, `Sized`, `Iterable[T]` et les modules standards documentés.
- [x] Les diagnostics d'héritage/mixins, `override` et `super` ont des tests
  négatifs exacts.
- [x] Les diagnostics de compatibilité orientent vers les compagnons recommandés
  (`Array.fill`, `Array.range`, `Set.fromArray`, `Option.some`,
  `Option.none`, etc.).
- [x] Les conventions runtime observables importantes sont documentées ou
  marquées comme internes.
- [x] `v0.1.0` a été tagué après validation sur `master`.

## Politique après 0.1

Les changements post-0.1 doivent rester orientés propreté et stabilité tant que
la surface publique n'est pas pleinement lisible :

- corrections de bug avec test ciblé ;
- corrections de diagnostic ;
- corrections de documentation ou d'exemples ;
- durcissement runtime/héritage sans élargir inutilement la surface publique ;
- nettoyage des APIs/documentations qui exposent encore des détails internes.

Toute extension majeure de surface publique (`Result[T]`, variance avancée,
linéarisation complète, GC, etc.) doit rester derrière une décision explicite et
un plan dédié.
