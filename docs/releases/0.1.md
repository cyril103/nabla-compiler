# Nabla 0.1 Scope

Ce document fixe le perimetre de Nabla 0.1. Il sert a eviter l'empilement de
fonctionnalites avant que le coeur du langage soit stable, documente et teste.

## Objectif

Nabla 0.1 doit permettre d'ecrire de petits programmes idiomatiques avec une
syntaxe Scala-like, un typage statique simple et une compilation native Linux
x86-64 via NASM/ELF.

Le but n'est pas encore d'etre un langage generaliste complet. Le but est un
socle fiable : fonctions, classes, heritage, mixins, collections de base,
lambdas et IO minimal.

## Fonctionnalites Incluses

### Langage

- Fonctions globales avec typage explicite.
- Variables locales `val` / `var`.
- Expressions `if` / `else if` / `else`.
- Boucles `while` et `for`.
- `match` simple avec wildcard final.
- Classes avec constructeur primaire.
- Champs et methodes.
- Heritage nominal simple via `extends`.
- Mixins via `with`, avec diagnostics de conflits.
- `override` obligatoire et signature strictement verifiee.
- `super` dans les methodes de classe.
- Hiérarchie racine Scala-like `Any` / `AnyVal` / `AnyRef`.
- Boxing runtime minimal des primitives vers `Any` / `AnyVal` pour préserver
  `toString()` dynamique sur les appels de fonctions/méthodes.
- Generiques simples invariants.
- Lambdas et types fonction `(A, B) => C`.
- Captures de lambda par valeur.

### Types De Base

- `Unit`
- `Bool`
- `Int`
- `Long`
- `Float`
- `Double`
- `Char`
- `String`

### Stdlib Publique Minimale

- `Array[T]`, `ArrayFill[T]`, `ArrayRange`.
- `Option[T]` et `OptionInt` tant que la specialisation primitive reste utile.
- `Set` actuellement disponible.
- Fonctions `strings`, `math`, `io` et `util/random` documentees.

La reference de surface est `docs/stdlib-api.md`.

### Backend Et Outillage

- Backend natif Linux x86-64.
- Generation NASM `elf64`.
- Edition de liens via `ld`.
- Diagnostics clairs quand `nasm` ou `ld` est absent.
- `make all-tests` vert.
- `make tooling-tests` vert.
- `make stdlib-docs` sans diff non commite.
- Compilation C++ stricte avec `-Wall -Wextra -Werror`.

## Non-Objectifs Pour 0.1

Ces sujets sont explicitement reportes :

- GC complet ou compactant.
- Backend autre que Linux x86-64 NASM/ELF.
- Unicode complet pour `String`/`Char`.
- Modules prives/export explicite au niveau langage.
- Traits/mixins avec linearisation Scala complete.
- Variance generique avancee.
- Typeclasses/implicites.
- Pattern matching exhaustif avance.
- `Result[T]`, `Map[K,V]` et collections riches hors besoins immediats.
- Optimisations agressives.
- ABI publique stable.

## Criteres De Sortie

Nabla 0.1 est pret quand :

1. La branche principale de developpement passe :

```bash
PATH=/opt/data/local/usr/bin:$PATH make all-tests
make examples
make tooling-tests
make stdlib-docs
git diff --exit-code docs/stdlib
g++ -std=c++17 -Wall -Wextra -Werror \
  src/main.cpp src/parser.cpp src/ast.cpp src/semantic_analyzer.cpp \
  src/ir.cpp src/ir_codegen.cpp src/runtime_asm.cpp -o /tmp/nablac-werror
```

2. `docs/language.md`, `docs/stdlib-api.md`, `docs/internals.md` et la
   reference HTML `docs/stdlib/` sont coherents.
3. Les exemples publics n'utilisent pas les helpers internes quand une facade
   publique existe.
4. Les erreurs utilisateur courantes ont des diagnostics lisibles :
   - type inconnu ;
   - methode/fonction inconnue ;
   - mauvaise arite ;
   - mauvais type d'argument ;
   - conflit d'heritage/mixin ;
   - `override` incorrect ;
   - `super` invalide ;
   - outil externe manquant.
5. Les conventions runtime observables sont documentees ou volontairement
   marquees comme internes.

## Roadmap Courte Vers 0.1

1. Nettoyer la documentation stdlib pour que la surface publique soit evidente.
2. Continuer a formaliser les conventions runtime qui restent implicites
   (header objet, vtables completes, erreurs runtime observables, limites
   memoire).
3. Extraire les gros blocs purs du compilateur : aliases stdlib, resolution
   heritage/mixins, helpers de types.
4. Ajouter des exemples idiomatiques centres sur `Array[T]`,
   `Array.fill[T]`, `Array.range`, `Array.rangeUntil`,
   `Option.some[T]` / `Option.none[T]`, classes, lambdas et IO.
5. Ameliorer les diagnostics d'heritage/mixins et garder un test negatif par
   classe d'erreur importante.
6. Faire une passe finale `make all-tests` + docs + exemples avant tag.

## Politique De Changements

Avant 0.1, les details internes peuvent changer librement. En revanche, toute
modification visible de la surface publique doit :

- etre decrite dans `docs/stdlib-api.md` ou `docs/language.md` ;
- ajouter ou adapter un test ;
- mettre a jour `docs/internals.md` si une convention runtime change ;
- etre commitee separement avec un message explicite.
